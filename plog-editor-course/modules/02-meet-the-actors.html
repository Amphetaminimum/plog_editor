Every good system is a cast, not a blob.

Meet the actors and what each one is responsible for

Think of the codebase like a small magazine studio

The studio has a front desk, an editor-in-chief, a storage room, and specialists for export and layout. If you know who does what, you can tell AI exactly where new logic belongs.

index.html The visible shell: toolbar, canvas, inspector, dialogs, and templates.
app.js The conductor that wires state, events, managers, and user actions together.
js/ Focused helper modules instead of one giant file.
doc-store.jsDocument library, metadata, restore and save behavior.
editor-render.jsTurns element data into live editable DOM nodes.
history-manager.jsUndo and redo snapshots plus operation history.
export-manager.jsDownloads raster images or standalone HTML.
storage.jsBrowser database helpers for docs and image blobs.

The system is split on purpose

🎛️

HTML shell

Holds the controls and empty mounting points so JavaScript has somewhere to plug in.

🧠

Central state

Keeps the source of truth for elements, selection, docs, zoom, and mode.

🧩

Managers

Each manager owns one slice of complexity instead of burying everything in one file.

🧭
Practical steering rule

If you want a new behavior that changes only zoom and responsive shell behavior, ask AI to edit shell-manager.js, not random places in app.js.

PLAIN ENGLISH

This file is not trying to do every job itself. It is importing specialists.

Each import name tells you the rough system map: layout, docs, rendering, history, shell, export, and storage.

That is a strong hint that the project has already moved away from “one giant script” thinking.
