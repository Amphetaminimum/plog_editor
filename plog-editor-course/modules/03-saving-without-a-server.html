How your plog survives without a server

The storage model is closer to a personal locker than a cloud drive

The app uses IndexedDB plus a tiny bit of localStorage. The big idea is simple: keep your real content locally, then remember which document you were using last.

📚

Docs store

Keeps the list of plogs, metadata like title and preview text, and the current doc ID.

🧱

State snapshot

Stores elements, zoom, theme, export settings, and other UI choices together.

🖼️

Asset blobs

Images live as binary blobs in a separate bucket so large media is handled cleanly.

Real code: the browser database is opened lazily

CODE 

const DB_NAME = "plog_editor_db";
const DB_STORE = "kv";
const ASSET_STORE = "assets";
let dbPromise = null;

function openDatabase() {
  if (dbPromise) return dbPromise;

  dbPromise = new Promise((resolve, reject) => {
    const request = indexedDB.open(DB_NAME, 1);
    request.onupgradeneeded = () => {
      const db = request.result;
      if (!db.objectStoreNames.contains(DB_STORE)) {
        db.createObjectStore(DB_STORE);
      }
      if (!db.objectStoreNames.contains(ASSET_STORE)) {
        db.createObjectStore(ASSET_STORE);
      }
    };
    request.onsuccess = () => resolve(request.result);
    request.onerror = () => {
      dbPromise = null;
      reject(request.error || new Error("indexeddb-open-failed"));
    };
  });

  return dbPromise;
}
PLAIN ENGLISH

The app names one browser database and two internal shelves: one for general key-value data and one for image assets.

It does not open the database over and over. It remembers the in-progress promise so repeated callers can reuse it.

If this is the first time the database is created, the app sets up the missing stores before regular reads and writes begin.

Design tradeoff

This is excellent for privacy and speed, but it also means there is no built-in cloud sync. If your browser storage is wiped, the app has no remote backup to rescue you.
