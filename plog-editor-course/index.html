function createElement(type, patch = {}) {

    },

  };
PLAIN ENGLISH

This function is the template factory. Every new thing you add starts here.

It looks at the current canvas layout and at the thing that came before, so new blocks can land in a sensible place.

Then it creates one base object with an ID, position, size, empty content, and default visual settings.

Nothing is visible yet. This is still just data waiting to be rendered.

This is the part that makes the app feel trustworthy.

How your plog survives without a server

The storage model is closer to a personal locker than a cloud drive

The app uses IndexedDB plus a tiny bit of localStorage. The big idea is simple: keep your real content locally, then remember which document you were using last.

📚

Docs store

Keeps the list of plogs, metadata like title and preview text, and the current doc ID.

🧱

State snapshot

Stores elements, zoom, theme, export settings, and other UI choices together.

🖼️

Asset blobs

Images live as binary blobs in a separate bucket so large media is handled cleanly.

Real code: the browser database is opened lazily

CODE 

const DB_NAME = "plog_editor_db";
const DB_STORE = "kv";
const ASSET_STORE = "assets";
let dbPromise = null;

function openDatabase() {
  if (dbPromise) return dbPromise;

  dbPromise = new Promise((resolve, reject) => {
    const request = indexedDB.open(DB_NAME, 1);
    request.onupgradeneeded = () => {
      const db = request.result;
      if (!db.objectStoreNames.contains(DB_STORE)) {
        db.createObjectStore(DB_STORE);
      }
      if (!db.objectStoreNames.contains(ASSET_STORE)) {
        db.createObjectStore(ASSET_STORE);
      }
    };
    request.onsuccess = () => resolve(request.result);
    request.onerror = () => {
      dbPromise = null;
      reject(request.error || new Error("indexeddb-open-failed"));
    };
  });

  return dbPromise;
}
PLAIN ENGLISH

The app names one browser database and two internal shelves: one for general key-value data and one for image assets.

It does not open the database over and over. It remembers the in-progress promise so repeated callers can reuse it.

If this is the first time the database is created, the app sets up the missing stores before regular reads and writes begin.

Real code: the app sanitizes and mirrors editable content back into state

CODE 

function setupNodeOnce(node) {
  const syncEditableItem = (editable) => {
    const host = editable.closest(".el");
    const current = host ? getElement(host.dataset.id) : null;
    if (!current) return;
    const cleanHtml = sanitizeEditableHtml(editable.innerHTML || "");
    if (editable.innerHTML !== cleanHtml) {
      editable.innerHTML = cleanHtml;
    }

    if (editable.classList.contains("content")) {
      current.html = cleanHtml;
      current.content = editable.innerText || "";
    } else if (editable.classList.contains("quote-content")) {
      current.html = cleanHtml;
      current.content = editable.innerText || "";
    }

    reflowAfterElement(current.id);
    render();
    saveSession();
  };
PLAIN ENGLISH

When a user edits visible content, the code finds the matching element object underneath that DOM node.

Before saving anything, it cleans the HTML so pasted junk or unexpected formatting does not poison the state.

Then it updates the content, reflows the layout, redraws the view, and persists the session.

Three clever patterns worth stealing for your own AI projects

🧼

Sanitize on the way in

User input is cleaned before it becomes trusted state. That prevents messy rich-text paste behavior from spreading.

📐

Reflow after edits

The app recalculates layout when text changes, so blocks keep stacking in a readable way.

🔄

Operation-aware undo

The history manager remembers more than raw snapshots. It also understands specific moves like insert, delete, and resize.

Aha: zoom and width are treated as separate concerns

CODE 

export function authoredCanvasWidthFromControls(widthMode, customWidthValue) {
  if (widthMode === "custom") {
    return clamp(Number(customWidthValue) || 1200, 480, 2400);
  }
  return clamp(Number(widthMode) || 1200, 480, 2400);
}

export function fitZoomRatioForStage({ stageClientWidth, paddingLeft, paddingRight, authoredWidth }) {
  const available = Math.max(180, stageClientWidth - paddingLeft - paddingRight - 8);
  return clamp(available / Math.max(1, authoredWidth), 0.08, 1);
}
PLAIN ENGLISH

First the app decides how wide the author wants the canvas to be in its own internal world.

Then it separately calculates how much that world should be scaled to fit the current screen.

That separation is why the app can support authored width choices and responsive zoom at the same time.

💡
Universal engineering lesson

Keep “what the content is” separate from “how it is currently displayed.” That single separation saves you from many responsive-layout bugs.

How export works, where the edges are, and what this codebase is ready for next

This app has two exit doors: raster image and standalone HTML

🖼️

Raster export

Turn the current canvas into PNG, JPG, or WebP so it can be posted like a long image.

🌐

HTML export

Package the current visual result into a self-contained HTML file that opens elsewhere.

🎛️

Appearance-aware output

The export layer can match the current theme or force a light or dark version.

Real code: exporting HTML means cloning the canvas and inlining the styles

CODE 

async function exportHtml() {
  flushRender();
  await hydrateAssetSources(getElements(), getAssetLoadToken());
  const clone = cloneCanvasForExport();
  const css = collectStylesText();
  const appearance = currentExportAppearance();
  const width = canvas.clientWidth;
  const height = canvas.offsetHeight;
  const html = `<!doctype html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>${escapeHtml(docName())}</title>
  <style>${css}</style>
</head>
<body class="${appearance === "dark" ? "theme-dark" : ""}" style="margin:0;background:${exportPalette().background};display:flex;justify-content:center;padding:24px;">
  <div style="position:relative;width:${width}px;min-height:${height}px;">${clone.outerHTML}</div>
</body>
</html>`;
}
PLAIN ENGLISH

Before exporting, the app forces the latest render and makes sure image sources are ready.

Then it clones the current canvas, grabs all styles from the page, and wraps the result inside a full HTML document string.

That is why the exported HTML can stand on its own: it carries both the visual markup and the CSS it needs.
