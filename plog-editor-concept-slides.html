Long Canvas Plog Editor

A local-first, browser-native editor for image-led stories: part Canva, part Notion, part long-image generator, built as a static frontend with no build step.

Positioning What this repo is trying to make
02
Product thesis

Not a conventional CMS. A composition tool for long-form visual posts.

The core idea from the PRD is clear: give creators one long canvas where images, headers, quotes, cards and text can be arranged freely, then export the result as a polished long image or HTML page.

Between tools

Positioned between Canva, Notion and long-image generators rather than copying any single one.

Image-led

The product assumes photos and visual rhythm lead the story, with text playing a supporting role.

Local-first

Documents live in the browser through IndexedDB, so the app stays simple, fast and private by default.

Experience Why long canvas matters
03
Core interaction

One vertical surface, flexible width, auto-expanding height.

  • Width presets like 1080px and 1200px target mobile sharing and blog reading.
  • Elements can be dragged and arranged like a visual essay instead of a rigid page builder.
  • The canvas becomes a pacing device for trips, coffee logs, photography journals and social posts.

Capabilities Current scope in the repo
04
Shipped direction

A focused editor surface with the right primitives already defined.

Element system

Text, image, divider, header, quote and card blocks form the current authoring vocabulary.

Typography controls

Font family, size, color, weight and spacing controls are part of the editing model.

Image treatment

Rotation, brightness, contrast, grayscale and frame styling show the app is designed for image work.

Export

PNG, JPG, WebP and HTML export make the output useful for social media and blog publishing.

Document workflow

Multiple local docs, rename/delete flows and lightweight persistence all support actual use.

No-build simplicity

Static frontend architecture keeps iteration cheap and lowers the barrier for product experiments.

Architecture How the app is structured
05
Frontend modules
index.html Main shell, templates, inspector and dialog mount.
styles.css Tokens, layout, editor surface and component styling.
app.js Editor state, render orchestration, events, export and UI coordination.
js/storage.js IndexedDB helpers for document data and binary assets.
Persistence snapshot
const DB_NAME = "plog_editor_db"; const DB_STORE = "kv"; const ASSET_STORE = "assets"; indexedDB.open(DB_NAME, 1); idbSet(key, value); idbGet(key); idbSetAsset(key, value); idbGetAsset(key);

This is the local-first contract in miniature: structured state in one store, heavier media assets in another, no backend required to keep editing sessions alive.

Potential Why this repo is worth attention
06
Product angle Niche but sharp

It targets creators who want polished long-form visuals, not generic document editing.

Execution angle Low overhead

Static files and browser storage make it easy to prototype, test and iterate without platform friction.

Design angle Format-native

The long canvas matches how travel logs, photo essays and social image posts are actually consumed.

Expansion angle Replaceable UI

The repo already notes that the data model and layout rules could survive a future iOS rewrite.

Roadmap What is unfinished but well-scoped
07
Current boundaries
  • Rich text modeling is not yet robust enough for production-grade editing.
  • Image crop and rotate handles still need more on-canvas precision.
  • Cloud sync, collaboration and batch export are explicitly out of scope for now.
Pragmatic next bets
  • Strengthen the text model and editing affordances.
  • Improve image manipulation to match the product's visual-first promise.
  • Decide whether mobile means responsive web, WebView shell, or native rewrite.
Closing What the concept says
08
Closing argument

This repo is compelling because it picks a format, a user and a workflow, then stays disciplined.

Instead of becoming a generic editor, it is trying to become the best place to compose long, image-led stories. That constraint gives the product its identity.

Clear thesis

Visual blogging deserves its own authoring surface.

Viable stack

HTML, CSS, JS and IndexedDB are enough for a convincing first product.

Portable future

The model can evolve to mobile later without throwing away the idea.
